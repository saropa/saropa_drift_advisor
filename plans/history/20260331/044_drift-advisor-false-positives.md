# 044 — Drift Advisor: False Positive Diagnostics

## Status: Open (Bug 1 fixed; Bug 2 fixed; Bug 3 fixed in v2.14.0)

## Problem

The Drift Advisor reports several diagnostic categories that produce false positives in projects with legitimate architectural reasons for the flagged patterns. These clutter the Problems panel and train developers to ignore real warnings.

---

## Bug 1: `no-foreign-keys` — false positive on intentionally isolated tables ✅ FIXED

**Actual severity:** Information (not hint)
**Category:** bestPractices
**Tables affected:** All 35+ Drift tables in the project

The advisor flags every table that has no `FOREIGN KEY` constraints. This is noisy for projects that:

1. Use UUID-based soft references across tables (common in Isar-to-Drift migration)
2. Store static/read-only data imported from bundled JSON (character sets, country data, vocabulary) where FK enforcement adds cost with zero benefit
3. Deliberately avoid FK constraints for performance (cascading deletes, insert ordering)

**Fix applied:** The check now only flags tables that have `_id` columns matching a known table name (e.g. `user_id` when a `users` table exists) but lack FK constraints. Tables referenced by other tables via inbound FKs are also skipped. This eliminates false positives on intentionally isolated tables (import caches, config stores, static data, audit logs) while still catching genuine missing `references()` calls. Plural matching handles `users`/`user`, `categories`/`category` (ies→y), etc.

## Bug 2: `empty-table` — runtime state reported as schema concern

**Actual severity:** Information (not hint)
**Category:** dataQuality
**Tables affected:** ~15 tables (static data, user data, system cache tables)

The advisor queries the development database and flags tables with 0 rows. This is a false positive because:

1. Static data tables are populated on first app launch from bundled assets — the dev DB on disk may not have been seeded
2. User data tables (coaching_sessions, connections, search_history, user_badges) start empty by design and fill as the user interacts with the app
3. System cache tables (wikimedia_births, wikipedia_articles, news_posts) are populated lazily on first network fetch

**Expected behavior:** `empty-table` should either:
- Not fire at all (row count is a runtime concern, not a schema concern)
- Be suppressible per-table via config
- Only fire if the table has been seeded by a migration and is still empty (indicating a failed migration)

**Workaround:** The rule CAN be suppressed globally:
- Right-click the diagnostic → "Disable `empty-table` rule" (quick fix)
- Or add to `.vscode/settings.json`:
  ```json
  { "driftViewer.diagnostics.disabledRules": ["empty-table"] }
  ```
- Or disable the entire category:
  ```json
  { "driftViewer.diagnostics.categories.dataQuality": false }
  ```

**Enhancement request:** Add per-table suppression or smarter heuristics (e.g. skip tables whose Dart IO layer has a seed/import method).

## Bug 3: `slow-query-pattern` reports advisor's own introspection queries ← CODE DEFECT

**Actual severity:** Warning (not error)
**Category:** performance
**Table affected:** `user_public_private_keys`

The advisor reports a slow query: `SELECT COUNT(*) AS c FROM (SELECT DISTINCT * FROM "user_p..."` at 124ms. However:

1. The IO layer (`user_public_private_keys_drift_io.dart`) does NOT use `COUNT(*)` or `DISTINCT` anywhere
2. The table holds at most 1 row (single-record pattern enforced in the IO layer)
3. The query IS generated by the advisor's own anomaly detector (`anomaly_detector.dart:287–289`, `_detectDuplicateRows`)

**Root cause:** In `router.dart:154`, all API handlers receive `_ctx.instrumentedQuery`, which records every SQL call into `queryTimings`. The anomaly detector (called via `router.dart:769` and `analytics_handler.dart:67`) runs through this same instrumented path, so its internal `SELECT COUNT(*)`, `SELECT DISTINCT *`, `PRAGMA table_info`, `PRAGMA foreign_key_list`, etc. are all captured in the timing buffer. The performance provider then picks these up and reports them as application slow queries.

**Fix:** The anomaly detector (and other introspection-only handlers) should use `_ctx.queryRaw` instead of `_ctx.instrumentedQuery` so their queries are not recorded in the performance buffer. Alternatively, add an `isInternal` flag to `QueryTiming` and filter internal queries out of the slow-query report.

**Workaround:** Clear performance data after an anomaly scan, or disable the performance category while anomaly detection runs.

---

## Suggested Improvements

1. **Per-table suppression:** Add inline `// drift_advisor:ignore no-foreign-keys` comments or a `drift_advisor.yaml` config for per-table/per-directory rule suppression
2. **Separate runtime from schema:** Distinguish between schema advisories (actionable at design time) and runtime state observations (not actionable without running the app)
3. **Fix introspection query leak (Bug 3):** Use `queryRaw` for internal analytics queries so they do not pollute the performance timing buffer
