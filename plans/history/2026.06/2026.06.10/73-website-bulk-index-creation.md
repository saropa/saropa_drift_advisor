# Feature 73: Website Bulk Index Creation

**Status: PLANNED** — split out of the archived GAP parity analysis (originally
WP-A05 / section 11 row "Bulk index creation"). The website applies index
suggestions one at a time; this adds multi-select batch apply with SQL preview.

Source: [GAP_FIT_PLAN.md (archive)](./history/2026.06/2026.06.10/GAP_FIT_PLAN.md) §11.

## Gap

| Surface | Capability            | State       |
| ------- | --------------------- | ----------- |
| W       | Bulk index creation   | **missing** |

## Tasks

- Add a multi-select control to the index suggestion list.
- Generate a merged SQL preview with per-index status.
- Apply as one action with partial-failure reporting.
- Provide rollback guidance (or generated rollback SQL when available).

## Exit criteria

- User can select 2+ suggestions, preview the merged SQL, apply in one action,
  and validate the resulting schema/indexes; partial failures are reported.

## Implementation Plan

### Current reality (correction to the original gap framing)

The website does **not** apply index suggestions "one at a time" — it does not
apply them at all. Suggestions are display-only: the SQL is rendered in a table
and clicking a cell copies it to the clipboard for manual paste
(`assets/web/tools-analytics.ts` → `initIndexSuggestions()` / `renderIndexData()`;
`#index-analyze` button in `lib/src/server/html_content.dart`). So this feature
adds an apply path *and* makes it batch from the start.

- Suggestions are generated by `IndexAnalyzer.getIndexSuggestionsList()`
  (`lib/src/server/index_analyzer.dart`); each is
  `{table, column, reason, sql, priority}`.
- Served at `GET /api/index-suggestions` via
  `AnalyticsHandler.handleIndexSuggestions()`
  (`lib/src/server/analytics_handler.dart`), registered in `_routeAnalyticsApi()`
  (`lib/src/server/router.dart`).
- A batch executor already exists for data mutations:
  `EditsBatchHandler.runValidatedBatchStatements()`
  (`lib/src/server/edits_batch_handler.dart`) wraps statements in
  `BEGIN IMMEDIATE; … COMMIT;` and reports per-statement failure via
  `BatchApplyStatementError` (index + statement + cause). **But** the validator
  (`SqlValidator.isSingleDataMutationSql()` in
  `lib/src/server/sql_validator.dart`) currently *rejects* `CREATE INDEX`.

### Phase 1 — Validator: allow CREATE INDEX

- Add `SqlValidator.isSingleCreateIndexSql()` matching
  `^CREATE\s+(UNIQUE\s+)?INDEX\s+(IF\s+NOT\s+EXISTS\s+)?…`, single-statement,
  no stacked DDL/PRAGMA. Keep it separate from the data-mutation validator so
  index DDL is gated by an explicit, narrowly-scoped check.

### Phase 2 — Merged SQL preview endpoint

- `POST /api/indexes/preview` (new constant in `server_constants.dart`): input
  `{indexSqls: [...]}`, output the validated, merged statement list (and reject
  reasons for any that fail validation). No DB write — preview only.

### Phase 3 — Batch apply with per-index status

- Reuse `runValidatedBatchStatements()` but swap in the new CREATE-INDEX
  validator; surface a per-index result array
  `[{index, sql, ok, error?}]` rather than failing the whole batch opaquely.
  Decide transaction semantics explicitly: all-or-nothing (current
  `BEGIN…COMMIT`) vs. best-effort per-index — recommend all-or-nothing first
  (simplest correct behavior), with partial-apply as a follow-up if requested.
- Route under `_routeWriteApi()` (write-enabled only, matching the existing
  `POST /api/edits/apply` gating).

### Phase 4 — Multi-select UI

- Extend `renderIndexData()` in `assets/web/tools-analytics.ts`: add a checkbox
  per suggestion, a selection count, a "Preview SQL" panel (calls Phase 2), and
  an "Apply selected" button (calls Phase 3, write-enabled only).
- Render the per-index result (green/red + error text) after apply; offer retry
  of only the failed rows.

### Exit gate

- Select 2+ suggestions, preview the merged SQL, apply in one action, see
  per-index success/failure, and confirm the created indexes exist in the live
  schema. Verify a deliberately bad suggestion reports its own failure without
  silently dropping the others.

## Finish Report (2026-06-10) — Phases 1–4

**Scope.** (A) Dart package (`lib/`, `test/`) + web assets (`assets/web/`). No Flutter app UI, no VS Code extension.

**Key decision — best-effort per index, not all-or-nothing.** The plan's Phase 3 floated all-or-nothing first, but the exit gate requires "a deliberately bad suggestion reports its own failure without silently dropping the others." All-or-nothing would roll the good indexes back, contradicting that. Since `CREATE INDEX` is independent and idempotent (with `IF NOT EXISTS`), apply runs each statement on its own and returns a per-index `{index, sql, ok, error?}` array. Documented at the top of `index_batch_handler.dart`.

**What changed.**
- **`lib/src/server/sql_validator.dart`** (Phase 1) — new `isSingleCreateIndexSql`: reuses the existing single-statement core (strips comments/strings, rejects multi-statement), requires a `CREATE [UNIQUE] INDEX [IF NOT EXISTS]` prefix, and rejects any stacked DDL/DML/PRAGMA keyword or a second `CREATE`. Kept separate from `isSingleDataMutationSql` so index DDL is gated by its own narrow check rather than loosening the data-mutation validator.
- **`lib/src/server/index_batch_handler.dart`** (new, Phases 2–3) — `handlePreview` (validate only, no write, works on read-only servers → `{valid, rejected}`) and `handleApply` (501 without `writeQuery`; best-effort per index → `{results, applied}`; invalid statements reported as failed without executing; `checkDataChange` fired only when ≥1 index was created).
- **`lib/src/server/server_constants.dart`** — `pathApiIndexesPreview/Apply` (+Alt) and the `indexSqls/valid/rejected/reason/results/index/applied` JSON keys.
- **`lib/src/server/router.dart`** — registered `IndexBatchHandler`; routed both endpoints under `_routeWriteApi` (preview there too, but it requires no `writeQuery`).
- **`assets/web/tools-analytics.ts`** (Phase 4) — `renderIndexData` gained an `interactive` flag adding a per-row checkbox, select-all, a selection count, **Preview SQL**, and **Apply selected** (shown only when `S.driftWriteEnabled`, else a read-only note). Checkboxes are keyed by row index and the SQL is read back from `lastIndexData` (never put in an attribute — `esc()` does not escape the double quotes index SQL is full of). Delegated container listeners survive re-render; preview/apply POST the selected SQL and render `{valid/rejected}` / per-index `{ok/error}` results, with a toast prompting a re-Analyze. Compare/history views stay non-interactive.
- **`assets/web/bundle.js`** — rebuilt via `npm run build:js` (committed; Dart consumers don't run Node).
- **`CHANGELOG.md`** — `[Unreleased]` Added entry.

**Testing.**
- **New `test/index_batch_handler_test.dart`** — 7 validator unit cases (plain/UNIQUE/IF NOT EXISTS/partial-WHERE/case-insensitive accepted; non-index DDL, DML, stacked, empty rejected) + 4 HTTP integration cases via the real `DriftDebugServer`: preview classifies valid/rejected with no write callback; apply → 501 without `writeQuery`; apply is best-effort (one DB failure + one validator rejection isolated, `applied` count correct, rejected SQL never reaches the DB); non-array body → 400.
- Audited existing tests referencing the touched symbols (router, validator, constants): `sql_validation_test.dart` (unchanged — `isReadOnlySql`/`isSingleDataMutationSql` untouched), handler/route tests still green.
- `dart analyze` clean; `dart test` → **571 passing**. `npm run typecheck:web` clean; `npm run build:js` rebuilt the bundle.

**l10n.** SKIPPED [web-not-Flutter] — the debug web viewer renders plain-English HTML and is not part of the Flutter ARB l10n catalog (no `l10n.*` getters exist for `assets/web/`). New strings match the surrounding convention in `tools-analytics.ts`.

**Outstanding.** None. All four phases complete; exit gate met (select 2+, preview, apply, per-index status, bad-statement isolation). Manual browser confirmation remains as a final verification step (see What to test).
